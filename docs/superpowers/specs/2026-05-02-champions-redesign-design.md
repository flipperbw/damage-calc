# Champions Damage Calculator — Frontend Redesign

**Date:** 2026-05-02
**Status:** Phase 1 design — approved, ready to plan
**Phase 2 (deferred):** Team builder with opponent recommendations

## Goal

Replace the existing vanilla-JS frontend with a modern, mobile-first, single-purpose damage calculator for the **Pokémon Champions** format. Single user (the repo owner). Optimized for fast, slick use mid-battle on a phone, while still working well on desktop.

## Non-goals

- All non-Champions modes (Honkalculate, OMs, Randoms, Singles, Index) are **removed**, not preserved.
- No backend, accounts, or sync. Local data only.
- No automated team-building, no opponent threat library beyond the existing `SETDEX_CHAMPIONS` data and an auto-tracked Recent Opponents list. Team builder is Phase 2 (separate spec).
- IVs, Tera, and Dynamax are **not exposed** — they don't apply to Champions. Megas do, and get a dedicated toggle.

## Stack

| Concern | Choice | Rationale |
|---|---|---|
| Framework | React 18 + TypeScript | Familiar, large ecosystem, fits a small SPA |
| Build | Vite | Fast dev server, simple config, native TS |
| Styling | Tailwind CSS | Utility-first matches mobile-first iteration |
| State | Zustand + persist middleware | Tiny, syncs to `localStorage` cleanly |
| Calc engine | `@smogon/calc` (existing `calc/` package, untouched) | Already implements Champions mechanics |
| Routing | None — single screen with tab state in store | Three tabs only; no URL routing needed |

The calc package stays a workspace dependency. The new app imports `calculate`, `Pokemon`, `Move`, `Field`, and `Generations` from `@smogon/calc`. `SETDEX_CHAMPIONS` currently lives at `src/js/data/sets/champions.js` (a global-style script). It will be ported to a typed module under `app/src/data/setdex-champions.ts` (or imported via a small shim) — exact path decided in the plan.

## Data model

```ts
type StatID = 'hp' | 'at' | 'df' | 'sa' | 'sd' | 'sp';

interface SavedMon {
  id: string;            // uuid, stable across renames
  species: string;       // canonical name from calc data, e.g. "Garchomp"
  buildName?: string;    // SETDEX_CHAMPIONS key, if a curated build was used
  item?: string;
  ability?: string;
  nature: string;        // default "Hardy"
  sps: Partial<Record<StatID, number>>; // 0..32 each, sum ≤ 66 (Champions limits)
  moves: [string, string, string, string]; // names; "" for empty
  isMega: boolean;       // toggle — when true, calc uses the mega forme
  currentHp?: number;    // raw HP; missing = full
  status?: 'Healthy' | 'Poisoned' | 'Badly Poisoned' | 'Burned' |
           'Paralyzed' | 'Asleep' | 'Frozen';
  boosts: Partial<Record<Exclude<StatID,'hp'>, number>>; // -6..+6
}

interface Team {
  id: string;
  name: string;
  format: 'singles' | 'doubles';
  mons: SavedMon[];      // 0..6
  createdAt: number;
  updatedAt: number;
}

interface FieldState {
  weather?: 'Sun' | 'Rain' | 'Sand' | 'Snow';
  terrain?: 'Electric' | 'Grassy' | 'Misty' | 'Psychic';
  isMagicRoom?: boolean;
  isWonderRoom?: boolean;
  isGravity?: boolean;
  // per-side
  yourSide: SideState;
  oppSide: SideState;
}

interface SideState {
  stealthRock?: boolean;
  spikes?: 0 | 1 | 2 | 3;
  reflect?: boolean;
  lightScreen?: boolean;
  auroraVeil?: boolean;
  tailwind?: boolean;
  protect?: boolean;
  leechSeed?: boolean;
  saltCure?: boolean;
  helpingHand?: boolean;
  isPowerTrick?: boolean;
  friendGuard?: boolean;
  isStatBoost?: boolean;     // "+1 all stats"
  isSwitching?: boolean;
  // The full set comes from calc/src/field.ts. The fields above are
  // those exposed in the Battle screen's expanded field drawer.
}

interface RecentOpponent {
  id: string;
  mon: SavedMon;         // snapshot at time of last use
  lastUsed: number;
  useCount: number;
}

interface AppState {
  teams: Team[];
  activeTeamId: string | null;
  activeMonIndex: number; // 0..5 within active team
  opponent: SavedMon | null;
  recentOpponents: RecentOpponent[]; // capped at 30, LRU eviction
  field: FieldState;
  notation: 'percent' | 'pixels'; // damage display preference
  tab: 'battle' | 'teams' | 'settings';
}
```

All state is persisted to `localStorage` under a single key (`champions-calc-v1`) via Zustand's `persist` middleware. Schema version stored alongside for future migration.

## Calc integration

For each render of the Battle screen, build the calc inputs from app state:

```ts
const gen = Generations.get(/* champions gen index */);
const attacker = new Pokemon(gen, mon.species, {
  item: mon.item, ability: mon.ability, nature: mon.nature,
  evs: mon.sps,                  // see Open Question #4: SP→calc input mapping
  boosts: mon.boosts,
  status: mon.status,
  curHP: mon.currentHp,
  // mega is applied via species override or item, depending on calc convention
});
const defender = new Pokemon(gen, opp.species, /* ...same */);
const field = new Field(buildField(state.field));
for (const moveName of mon.moves) {
  const move = new Move(gen, moveName);
  const result = calculate(gen, attacker, defender, move, field);
  // result.range, result.kochance(), result.desc()
}
```

KO/priority/ability flags shown on each move row come from `result.kochance()`, `move.priority`, and curated lookups against ability/item names (e.g. show "Sturdy" warning when defender is at full HP and a move would otherwise OHKO).

Mega evolution: when `mon.isMega` is true, the calc operates on the Mega forme of the species. Implementation detail: most likely we set the mon's species to e.g. `"Garchomp-Mega"` (forme suffix already used by `SETDEX_CHAMPIONS`) at calc time only, leaving the saved species canonical. Validated against `Pokemon.getForme` — if that helper covers it, use that.

## Screens

### Battle (default tab)

Single scrollable column on mobile. Top → bottom:

1. **Top bar** — active team pill (tap → quick switcher), settings cog.
2. **Field-state pills** — condensed, active ones glow. Inactive shown as faint placeholders ("+ Hazards", "+ Screens"). Tap any pill to expand a drawer with full per-side controls (every flag in `Field` for Champions).
3. **Team carousel** — your 6 mons as square slots with mini HP bars. Active mon outlined in violet glow. Tap to switch active. Long-press to edit. Fainted slots dimmed.
4. **Active mon card (you)** — sprite, name, types, editable HP slider with raw + percentage, status/ability/item/nature/boosts as chips, **Mega toggle** when species has a mega forme.
5. **Your moves block** — 4 move rows. Each shows: type badge, move name, KO tag (OHKO / 2HKO / N% chance to OHKO), damage % range (large), raw damage range (small).
6. **Speed/priority divider** — "⚡ You +47" or "⚠ They +N", plus orange flag for any priority moves either side has that flip the order ("⚠ Sucker Punch flips order").
7. **Opponent card** — dashed border to signal "not your team". All chips editable in place. Species name doubles as picker trigger. **HP shown in % only** for the opponent (per Champions in-game UI). Your own active mon's HP card shows both raw + %. Mega toggle if applicable. Recent Opponents pinned to top of species picker.
8. **Their moves block** — same row format as yours, damage to you, KO tag, priority flag.

### Teams

- Top bar: "Teams" title, ⊕ new team button.
- List of team cards. Active team highlighted. Each shows: name + emoji, format badge (singles/doubles), 6 mon slots, ⋯ menu (rename, duplicate, export to PSDM-style text, delete).
- Tap a team → makes it active and switches to Battle.
- Tap a mon slot → opens Mon Editor for that mon. Empty slot → opens species picker, then editor.
- "+ New Team" CTA.
- **Recent Opponents** section beneath: list of last 30 opponents with sprite + name + build + use count + ✕ to remove individually.

### Settings

- Notation: percent / pixels (48ths)
- Theme: Glass Dark (default). Light theme deferred — design is dark-only at launch.
- Import / Export all data as JSON.
- Clear recent opponents.
- Reset all data (with confirm).
- Build version, link to repo.

### Mon Editor

Bottom sheet (mobile) / right-side drawer (desktop ≥900px). Sections, top to bottom:

1. **Hero** — sprite, name (tap to swap species), types.
2. **Build dropdown** — lists every entry under `SETDEX_CHAMPIONS[species]`. Selecting one auto-fills item/ability/nature/SPs/moves. Any subsequent edit to those fields clears `buildName` (the dropdown shows "Custom" until the user picks a curated set again).
3. **Item / Ability / Nature** — three select rows.
4. **Mega toggle** — only shown when species has a mega forme. When on, stat/ability/type previews update to the mega forme.
5. **SP grid** — 6 cells (HP / Atk / Def / SpA / SpD / Spe). Each cell shows: stat label, computed total stat, "+N SP" allocation, mini bar to 32. Tap to edit (numeric stepper). Header shows running total "/ 66". Over-allocation shown red and blocks save.
6. **Moves** — 4 rows, each opens a move picker.
7. **Save** — large primary button pinned to the bottom of the sheet. Cancel via swipe-down / drawer close.

  "version": 1,
  "state": {
    "teams": [...],
    "activeTeamId": "...",
    "activeMonIndex": 0,
    "opponent": {...} | null,
    "recentOpponents": [...],

Migration strategy: on load, if `version < currentVersion`, run a sequence of versioned migrators. v1 launches as-is; future schema changes bump and add a migrator.

## Test strategy

- **Calc adapter** unit-tested against fixtures: known mon/move/field combos with expected damage ranges. Highest-value tests because the calc package is already trusted; the bug surface is in the adapter that translates app state to calc inputs.
- **Store reducers / actions** unit-tested with Vitest: team CRUD, recents LRU eviction, SP allocation validation (≤32/stat, ≤66 total).
- **Migration runners** unit-tested with snapshot inputs.
- **Component tests** (Vitest + Testing Library) only for the editor SP grid (validation visible to user) and the species picker (recents pinning behavior).
- **Playwright E2E** for durable golden-path coverage: load app → create team → add mons via build dropdown → set up opponent → verify damage values render and KO tags update on edit → recents persist across reloads. These tests live in `app/e2e/` and run in CI. Add tests as features land; don't pre-scaffold before there's something worth testing.
- **Ad-hoc verification** during development uses the Playwright MCP and Chrome DevTools MCP rather than manual smoketests. Required before claiming any UI change "works."

## Open implementation questions

These don't block the spec but will surface in the plan:

1. **Mega forme resolution** — confirm whether passing `species: "Garchomp-Mega"` to `new Pokemon()` is the calc's expected interface, vs. setting the mega stone item and letting calc infer. Decide in the implementation plan after a small spike.
2. **Champions generation index** — confirm which `Generations.get(N)` value corresponds to Champions. Read from `calc/src/data/index.ts` during planning.
3. **HP slider granularity** — drag steps of 1% on mobile feels right, but raw HP edge cases (Sturdy at exactly 1 HP) may need the raw input too. Shipping with both controls visible.
4. **SP → calc input mapping** — the existing UI has SP inputs but the `@smogon/calc` `Pokemon` constructor takes `evs`. Confirm whether SPs map 1:1 onto the `evs` slot for Champions mechanics, or whether the Champions calc path uses a different field. Read `calc/src/mechanics/champions.ts` and `calc/src/pokemon.ts` during planning to confirm.
